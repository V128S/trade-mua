# Информативные поля оформления заказа

**Дата:** 2026-06-02
**Статус:** утверждён дизайн

## Проблема

Текущая форма оформления заказа (`src/components/cart/CheckoutForm.tsx`) собирает мало
структурированных данных:

- адрес доставки — одно свободное поле «Нова Пошта» (`novaPoshta`);
- телефон (префилл из профиля);
- комментарий.

Имя получателя в форме вообще не собирается — в админке берётся из `profiles.full_name`.
Телефон при сохранении заказа «подмешивается» в `notes` (нет отдельной колонки).
В результате у директора/админа в карточке заказа нет чистых, раздельных данных получателя
и адреса доставки.

## Цель

Сделать поля оформления заказа информативными и раздельными: **имя, фамилия, телефон,
город, отделение Новой Почты** — как отдельные поля формы и отдельные колонки в БД,
с понятным отображением в админке.

## Решения (утверждены)

- **Хранение:** отдельные колонки в таблице `orders` (вариант A).
- **Новая Почта:** простые текстовые поля (без интеграции с API НП).
- **Обязательность:** имя, фамилия, телефон, город, отделение — обязательны; комментарий — нет.

## Объём работ

### 1. Миграция БД (`orders`)

Добавить 5 nullable-колонок (nullable — чтобы не сломать существующие заказы):

| Колонка | Тип | Назначение |
|---|---|---|
| `recipient_first_name` | text | Имя получателя |
| `recipient_last_name` | text | Фамилия получателя |
| `recipient_phone` | text | Телефон получателя |
| `city` | text | Город |
| `nova_poshta_branch` | text | Номер/название отделения НП |

Существующая колонка `nova_poshta_address` **сохраняется** для обратной совместимости
со старыми заказами (в админке — фоллбэк-отображение). Для новых заказов она заполняется
производным значением `"{city}, {branch}"`.

### 2. Форма `CheckoutForm.tsx`

Порядок и поведение полей:

1. **Имя** + **Фамилия** — в ряд (grid 2 колонки на ≥sm), оба `required`.
2. **Телефон** — `required`, плейсхолдер `+380…`, префилл из профиля.
3. **Город** + **№ отделения Новой Почты** — в ряд (grid 2 колонки на ≥sm), оба `required`.
4. **Комментарий** — `textarea`, необязателен.

Префилл имени/фамилии — best-effort: `profile.full_name` разбивается по первому пробелу
(`first = до первого пробела`, `last = остаток`). Если `full_name` пуст — поля пустые.

Состояние компонента: добавить `firstName`, `lastName`, `city`, `branch`; удалить
объединённое `novaPoshta`. Стили полей (`field`, `labelCls`) переиспользуются как есть.
Сетка из двух колонок реализуется через `grid grid-cols-1 sm:grid-cols-2 gap-4`.

### 3. Префилл (страница `checkout/page.tsx`)

Запрос профиля расширить с `phone` до `full_name, phone`. Передавать в `CheckoutForm`
`defaultPhone` и новый проп `defaultFullName` (для разбиения на имя/фамилию).

### 4. Серверный экшен `placeOrder` (`src/lib/cart/actions.ts`)

- Расширить `PlaceOrderInput`: `firstName`, `lastName`, `city`, `branch` (заменяя `novaPoshta`).
- При вставке заказа писать новые колонки: `recipient_first_name`, `recipient_last_name`,
  `recipient_phone` (= `input.phone.trim()`), `city`, `nova_poshta_branch`.
- `notes` теперь содержит **только** комментарий (перестать подмешивать телефон).
- `nova_poshta_address` заполнять как `"{city}, {branch}"` (обратная совместимость).
- Валидация на сервере: обязательные поля не пустые после `trim()`; иначе вернуть
  `{ error: '...' }` (украиноязычное сообщение в стиле существующих).

### 5. Админка `OrdersTable.tsx`

Блок данных получателя в карточке заказа:

- **Имя получателя:** `recipient_first_name + recipient_last_name` из заказа; фоллбэк —
  `profile.full_name`; иначе «Анонім».
- **Телефон:** `recipient_phone` из заказа (фоллбэк — `profile.phone`), кликабельный `tel:`.
- **Адрес:** отдельные строки «Місто» (`city`) и «Відділення НП» (`nova_poshta_branch`);
  для старых заказов без этих колонок — фоллбэк на `nova_poshta_address`.

Поиск (`query`) дополнительно учитывает `recipient_first_name`/`recipient_last_name`.

Тип `Order` расширяется новыми колонками автоматически через обновлённый `Database` тип.

### 6. i18n (`src/messages/ru.json`, `src/messages/uk.json`)

Namespace `checkout` — добавить ключи:

- `labelFirstName`, `labelLastName`, `labelCity`, `labelBranch`;
- плейсхолдеры: `placeholderCity`, `placeholderBranch` (например `№ відділення`);
- сообщения валидации при необходимости (`errorRequired`).

`labelNovaPoshta`/`placeholderNovaPoshta` заменяются на `labelBranch`/`placeholderBranch`.
Тексты — на украинском (UI-язык сайта), затем русские эквиваленты в `ru.json`.

### 7. Типы (`src/lib/types/database.types.ts`)

Добавить новые колонки `orders` в `Row` и `Insert` (`recipient_first_name`,
`recipient_last_name`, `recipient_phone`, `city`, `nova_poshta_branch` — все `string | null`,
в `Insert` опциональны).

## Вне объёма (YAGNI)

- Интеграция с API Новой Почты (автоподсказка городов/отделений).
- Изменение схемы `profiles` (раздельные имя/фамилия в профиле) — префилл делается из `full_name`.
- Сохранение адреса доставки в профиль для повторного использования.

## Совместимость / риски

- Колонки nullable — старые заказы не ломаются; админка показывает их через фоллбэк
  на `nova_poshta_address`/`profiles`.
- Миграция применяется к боевому проекту Supabase напрямую (как в этом репозитории принято);
  деплой кода — обычным push в `main`.
